Asana API – LLM-optimierte Zusammenfassung (Plain Text)

1. Überblick
Die Asana API ist eine REST-basierte HTTP-API.
Hauptzweck: Projekte, Tasks, Workspaces, Teams, Users, Tags etc. lesen und verändern.
Basis: JSON over HTTPS, meist unter /api/1.0/

Basis-URL (Cloud):
https://app.asana.com/api/1.0/

Alle Ressourcen hängen unter dieser Basis-URL.

2. Authentifizierung

Asana verwendet Personal Access Tokens (PAT) oder OAuth2.

2.1. Personal Access Token (einfachste Variante)
Header:
Authorization: Bearer <PERSONAL_ACCESS_TOKEN>

Beispiel:
GET https://app.asana.com/api/1.0/users/me
Authorization: Bearer 1/1234567890abcdef...

2.2. OAuth2 (3-legged)
- Standard OAuth2-Flow mit Client ID, Client Secret, Redirect URI.
- Nach erfolgreichem Flow erhält man ein Access Token, das genauso im Bearer-Header verwendet wird.

3. Allgemeines Request-/Response-Muster

Request:
- HTTP-Methoden: GET, POST, PUT, DELETE
- Content-Type für Bodies: application/json
- Query-Parameter z. B. für Pagination und Feldselektion (opt_fields)

Response:
- JSON-Struktur mit Wurzel-Property "data"
- Bei Listen zusätzlich "next_page" für Pagination

Beispiel-Response (Liste):
{
  "data": [ ... ],
  "next_page": {
    "offset": "xyz",
    "path": "/api/1.0/tasks?offset=xyz",
    "uri": "https://app.asana.com/api/1.0/tasks?offset=xyz"
  }
}

4. Wichtige Ressourcen / Endpunkte

4.1. Nutzer

Eigener User:
GET /users/me

User-Liste:
GET /users

User nach ID:
GET /users/{user_gid}

4.2. Workspaces (Organisationen)

Workspaces auflisten:
GET /workspaces

Workspace nach ID:
GET /workspaces/{workspace_gid}

4.3. Projekte

Projekte in einem Workspace:
GET /projects?workspace={workspace_gid}

Projekt nach ID:
GET /projects/{project_gid}

Projekt erstellen:
POST /projects
Body (Beispiel):
{
  "data": {
    "name": "Neues Projekt",
    "workspace": "{workspace_gid}"
  }
}

Projekt aktualisieren:
PUT /projects/{project_gid}

4.4. Tasks (Aufgaben)

Tasks in einem Projekt:
GET /tasks?project={project_gid}

Tasks, die einem User zugewiesen sind:
GET /tasks?assignee={user_gid}&workspace={workspace_gid}

Task nach ID:
GET /tasks/{task_gid}

Task erstellen:
POST /tasks
Body:
{
  "data": {
    "name": "Neue Aufgabe",
    "projects": ["{project_gid}"],
    "workspace": "{workspace_gid}",
    "assignee": "{user_gid (optional)}"
  }
}

Task aktualisieren:
PUT /tasks/{task_gid}

Task löschen:
DELETE /tasks/{task_gid}

Subtasks eines Tasks:
GET /tasks/{task_gid}/subtasks

Kommentare (Stories) eines Tasks:
GET /tasks/{task_gid}/stories

Kommentar hinzufügen:
POST /stories
Body:
{
  "data": {
    "text": "Kommentar",
    "target": "{task_gid}"
  }
}

4.5. Tags

Tags in Workspace:
GET /tags?workspace={workspace_gid}

Tag nach ID:
GET /tags/{tag_gid}

Tag auf Task anwenden:
POST /tasks/{task_gid}/addTag
Body:
{
  "data": {
    "tag": "{tag_gid}"
  }
}

4.6. Teams (innerhalb einer Organisation)

Teams einer Organisation:
GET /organizations/{workspace_gid}/teams

Team-Projekte:
GET /teams/{team_gid}/projects

5. Felder auswählen (opt_fields)

Asana erlaubt, zu definieren, welche Felder zurückgegeben werden.

Beispiel:
GET /tasks/{task_gid}?opt_fields=name,assignee,status,due_on

Das verringert Payload-Größe und klärt für das LLM, welche Attribute relevant sind.

6. Pagination

Asana verwendet ein Offset-basiertes Pagination-Modell.

- Query-Parameter: limit, offset
- Response.next_page.offset enthält den Wert für den nächsten Request.

Beispiel:
GET /tasks?project={project_gid}&limit=50

Response:
{
  "data": [...],
  "next_page": {
    "offset": "xyz",
    ...
  }
}

Nächste Seite laden:
GET /tasks?project={project_gid}&limit=50&offset=xyz

7. Fehler-Handling

Typische HTTP-Statuscodes:
- 200: OK
- 201: Created
- 400: Bad Request
- 401: Unauthorized (Token fehlt oder ist ungültig)
- 403: Forbidden (keine Berechtigung)
- 404: Not Found
- 429: Rate Limit erreicht
- 500: Server Error

Fehler-Body:
{
  "errors": [
    {
      "message": "...",
      "help": "https://asana.com/developers/documentation/..."
    }
  ]
}

8. Webhooks (nur kurz angerissen)

Asana unterstützt Webhooks für Events an Ressourcen (z. B. Tasks).
- POST /webhooks zum Anlegen
- Events werden per HTTP POST an Callback-URL zugestellt.
LLMs können Webhooks erwähnen, aber in der Regel richtet ein Backend sie ein.

9. Beispiel: Vollständiger HTTP-Request

GET https://app.asana.com/api/1.0/tasks?project={project_gid}&limit=10
Authorization: Bearer <TOKEN>
Accept: application/json

Mögliche Response:
{
  "data": [
    {
      "gid": "1234567890",
      "name": "Task 1",
      "resource_type": "task"
    }
  ],
  "next_page": null
}

10. Hinweise speziell für LLM-Agenten

- Immer Basis-URL + /api/1.0/ verwenden.
- Authorization Header im Bearer-Format setzen.
- Wichtige Top-Level-Objekte: workspaces, projects, tasks, users, tags, teams.
- Ressourcenerkennung: Viele Beziehungen laufen über GIDs (globally unique IDs, Strings).
- Bei Listen-Requests Pagination über next_page beachten.
- Felder gezielt mit opt_fields auswählen, um saubere, kleinere Responses zu bekommen.
- Mutationen (POST/PUT/DELETE) immer innerhalb einer "data" Hülle im Body senden.
- Asana ist stark auf Workspaces/Organisationen ausgerichtet – fast jede Abfrage braucht eine Workspace-ID.
