Monday.com API – Vollständige Objektübersicht & Request-Guide (LLM-Optimiert)

========================================================================
1. GRUNDLAGEN
========================================================================

Die Monday.com API ist vollständig GraphQL-basiert.
Es gibt NUR EINEN Endpunkt:

POST https://api.monday.com/v2

HEADERS (immer notwendig):
Authorization: 
Content-Type: application/json

REQUEST BODY FORMAT:
{
  "query": "",
  "variables": { ...optional }
}

Alle Abfragen und Mutationen laufen ausschließlich über GraphQL.


========================================================================
2. ALLE HAUPT-OBJEKTE, DIE ABGEFRAGT WERDEN KÖNNEN
========================================================================

Ein LLM-Agent kann die folgenden Objekte ansprechen:

✔ boards                 (Workspaces / Projekte)  
✔ groups                 (Gruppen innerhalb eines Boards)  
✔ items                  (Tasks / Zeilen)  
✔ subitems               (Subtasks)  
✔ columns                (Metadaten-Spalten eines Boards)  
✔ column_values          (Inhalte der Items pro Spalte)  
✔ updates                (Kommentare / Activity Feed Messages)  
✔ users                  (User Accounts)  
✔ teams                  (Teams in einem Account)  
✔ workspaces             (Neue API-Funktion, nicht immer sichtbar)  
✔ webhooks               (Echtzeit-Trigger)  
✔ tags                   (Labels)  
✔ account                (Account-level Informationen)
✔ rate_limits            (Abfragen zum aktuellen API-Verbrauch)

Zusätzlich generisch:
✔ assets (Datei-Upload/Download via Mutations)
✔ notifications
✔ activity_logs

NOTIZ: Die tatsächliche Abfragemenge hängt vom GraphQL-Schema ab:
https://api.monday.com/v2/get_schema?format=sdl


========================================================================
3. ANFRAGEN: WIE SIE FORMATIERT SEIN MÜSSEN
========================================================================


---------------------------------------
3.1 Grundstruktur eines Requests
---------------------------------------

POST https://api.monday.com/v2
Headers: Authorization + Content-Type

Body:
{
  "query": "query { boards { id name } }"
}

---------------------------------------
3.2 Komplexere Queries
---------------------------------------

Ein Query darf verschachtelte Felder enthalten. Beispiel:

{
  boards(limit: 20) {
    id
    name
    state
    workspace {
      id
      name
    }
    groups {
      id
      title
    }
    items(limit: 50) {
      id
      name
      created_at
      column_values {
        id
        text
        value
      }
    }
  }
}


========================================================================
4. DETAILLIERTE OBJEKTÜBERSICHT MIT QUERY-BEISPIELEN
========================================================================


---------------------------------------
4.1 BOARDS
---------------------------------------

Query (alle Boards):

{
  boards {
    id
    name
    description
    state
    board_kind
    owner {
      id
      name
    }
    workspace {
      id
    }
  }
}

Board-spezifisch:
- id
- name
- state (active / archived / deleted)
- description
- type / kind (public / private / shareable)
- workspace
- groups
- items
- permissions


---------------------------------------
4.2 GROUPS
---------------------------------------

Query:

{
  boards(ids: [12345]) {
    groups {
      id
      title
      color
      archived
    }
  }
}

Attribute:
- id
- title
- color
- archived
- deleted
- position


---------------------------------------
4.3 ITEMS (Tasks)
---------------------------------------

Query:

{
  items(ids: [98765]) {
    id
    name
    created_at
    creator_id
    group {
      id
    }
    column_values {
      id
      text
      value
    }
  }
}

Attribute:
- id
- name
- created_at
- updated_at
- group
- board
- column_values[]
- subitems[]


---------------------------------------
4.4 SUBITEMS
---------------------------------------

{
  items(ids: [98765]) {
    subitems {
      id
      name
      column_values {
        id
        text
      }
    }
  }
}


---------------------------------------
4.5 COLUMNS
---------------------------------------

{
  boards(ids: [12345]) {
    columns {
      id
      title
      type
      settings_str
    }
  }
}

Column Types u.a.:
- status
- text
- long_text
- date
- people
- numbers
- dropdown
- tags
- timeline
- files
- link
- formulas
- mirror
- dependency


---------------------------------------
4.6 COLUMN VALUES
---------------------------------------

{
  items(ids: [98765]) {
    column_values {
      id
      text
      value  // JSON-String
    }
  }
}

value enthält das Roh-JSON für komplexe Spalten.

========================================================================
5. MUTATIONEN (ERSTELLEN / ÄNDERN)
========================================================================


---------------------------------------
5.1 Item erstellen
---------------------------------------

mutation {
  create_item(
    board_id: 12345,
    item_name: "New Task",
    group_id: "topics"
  ) {
    id
  }
}


---------------------------------------
5.2 Spaltenwert ändern (simple)
---------------------------------------

mutation {
  change_simple_column_value(
    item_id: 98765,
    board_id: 12345,
    column_id: "status",
    value: "Done"
  ) {
    id
  }
}


---------------------------------------
5.3 Spaltenwert ändern (JSON)
---------------------------------------

mutation {
  change_column_value(
    board_id: 12345,
    item_id: 98765,
    column_id: "date",
    value: "{\"date\":\"2025-01-10\"}"
  ) {
    id
  }
}


---------------------------------------
5.4 Update (Kommentar) erstellen
---------------------------------------

mutation {
  create_update(
    item_id: 98765,
    body: "Hello world!"
  ) {
    id
  }
}

      { "id": "12345", "name": "Marketing", "state": "active" }
    ]
  }
}


========================================================================
7. WICHTIGE HINWEISE FÜR LLM-AGENTEN
========================================================================

- Immer POST /v2 verwenden.
- Queries müssen vollständig sein (GraphQL Feldliste).
- API reagiert nur auf gültige GraphQL-Felder.
- Column Values sind dynamisch pro Board.
- Immer complexity limits beachten.
- Bei Unsicherheit zuerst get_schema abfragen.
- Für Datenermittlung: zuerst boards → groups → items → column_values.
